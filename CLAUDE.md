# CLAUDE.md

Нотатки для Claude Code при роботі з цим репозиторієм. Проєкт описаний у README.md — тут лише те, що корисно знати для внесення змін.

## Що це за код

Три великих однофайлових HTML-застосунки (Tailwind CDN + inline `<style>`/`<script>`, без збірки):
- `index.html` — shell з PIN-входом і табами, вантажить інші два модулі в iframe.
- `osbb/index.html` (~2600 рядків) — Журнал: чергування, сміття, диспетчер, чат. Акцент — зелений (`#34c759` світла тема / `#30d158` темна).
- `sklad/index.html` (~2100+ рядків) — Склад: товари, видача, інвентаризація. Теж зелений акцент (уніфіковано в липні 2026, раніше був синій).

Немає системи збірки, немає npm-залежностей у фронтенді — усе редагується прямо у величезних HTML-файлах через Edit tool.

## Усталені патерни дизайн-системи (не винаходити заново)

- **Кастомний `<select>`**: нативний select рендериться браузером як квадратне непіддатливе спливне меню. `enhanceSelect(select)` / `refreshEnhancedSelect(select)` тепер живуть в одному місці — `shared/enhance-select.js` (звичайний `<script src>`, без бандлера), підключені і в journal, і в sklad. Не дублювати їх назад inline — smoke-check це ловить. Обгортає прихований `<select>` кастомною заокругленою панеллю (`.custom-select-wrap/-btn/-panel/-option`), з підтримкою клавіатури (Enter/Space/стрілки/Escape) і опційним пошуком через `data-searchable="1"`. Якщо десь ще лишився голий `<select>` — використовувати той самий патерн, не робити новий.
- **Спільні CSS/JS без бандлера**: `shared/ui.css` (тултіпи `[data-tip]`, `prefers-reduced-motion`) і `shared/enhance-select.js` — єдине місце для того, що реально ідентичне між модулями. Не все дублювання варто виносити сюди: якщо модулі розійшлись у поведінці (як було з `enhanceSelect` до липня 2026, коли Sklad додав пошук), спершу звести докупи функціонально, а не сліпо взяти першу-ліпшу копію.
- **Тултипи**: `[data-tip="текст"]` замість `title=` — CSS-псевдоелемент, заокруглений темний пільс замість системного. Клас `.tip-up` — якщо кнопка всередині `overflow:hidden` контейнера (напр. в таблиці), щоб підказка відкривалась вгору і не обрізалась.
- **Skeleton-заглушки завантаження**: клас `.skel`/`.skel-bar` (shimmer-анімація) замість тексту "Завантаження..." — використовується як статичний початковий HTML у `<tbody>`, перезаписується реальним рендером.
- **Плаваюча нижня навігація** (мобільна): `#bottom-nav` / `.bottom-nav` — не суцільна смуга по краю, а картка з відступом `12px` з усіх боків, `border-radius: 26px`, тінь. У osbb позиціювання йде через Tailwind-класи (`fixed bottom-0 left-0 right-0`), тому кастомні CSS-правила для нього **обов'язково з `!important`**, бо мають перебити Tailwind-утиліти.
- **SVG-іконки в osbb**: повторювані іконки (журнал/графіки/сміття/диспетчер/чат) винесені у `<symbol>` спрайт на початку `<body>` (`#i-journal`, `#i-charts`, ...), використовуються через `<svg><use href="#i-..."/></svg>`. Не плодити нові inline SVG для тих самих іконок — додавати `<symbol>` і `<use>`.
- **Фокус з клавіатури**: `outline:none` було глобально вбито без заміни — тепер скрізь є `:focus-visible { outline: 2px solid ... }`. Не видаляти ці правила.
- **`prefers-reduced-motion`**: є `@media (prefers-reduced-motion: reduce)` у всіх трьох файлах — глушить анімації/переходи.

## Супабейс (єдиний проєкт з липня 2026)

- **Один Supabase-проєкт** (`vkwkyhjjjmcpmiakxohw`, назва в Dashboard — "Sklad") для журналу й складу разом. До липня 2026 було два окремі проєкти; окремий проєкт журналу (`jpwelaerkkdblpmkueid`) видалено після перенесення даних — **більше не існує, не намагатись до нього звертатись чи його відновлювати.**
- Актуальна схема — виключно в `sklad/supabase/*.sql`, файли пронумеровані (`001_...` → `005_...`) і виконуються по порядку. `sklad/supabase/005_merge_osbb_journal.sql` — це весь журнал (`schedule`/`garbage`/`dispatcher`/`chat`/`photos` + PIN-и + RPC).
- `supabase/*.sql` (корінь репо) — **історичний архів видаленого проєкту журналу**, не актуальна схема. Не редагувати як робочі міграції, не радити користувачу їх виконувати на новому проєкті.
- **Жорстке правило**: якщо змінюєш живу схему Supabase напряму (через MCP tools, Dashboard SQL Editor тощо) — **у тому ж коміті** додай або онови відповідний файл у `sklad/supabase/*.sql` з тим самим DDL. Без цього репозиторій перестає бути достовірним джерелом істини щодо реальної схеми — так уже сталося один раз: тригер `trg_notify_low_stock` і зберігання Telegram-токена в таблиці `telegram_config` існували в живій базі, але не мали жодного SQL-файлу в git.
- Журнал і склад мають **окремі auth-таблиці навмисно**: `osbb_app_auth` (два PIN — вхід і скидання) vs `app_auth` складу (один PIN). Не намагатись "уніфікувати" авторизацію без явного запиту користувача — це архітектурне рішення, не забутий рефакторинг.

## Специфічно для Складу

- Товар можна позначити `is_internal` (внутрішнє використання/хознужди) — не входить у "баланс складу" на сторінці Статистика. Перемикається іконкою в рядку товару (`toggleInternal(id, val)`), фільтрується пілками "Без внутрішніх" / "Тільки внутрішні" (взаємовиключні, `hideInternal`/`onlyInternal`).
- **Облік вартості товарів додано як опційний**: `sklad/supabase/004_add_price_tracking.sql` додає `price_unit`, `price_source`, `price_url`, `price_checked_at`, `price_confidence` до `inventory_items`. `sklad/supabase/functions/fetch-item-prices` шукає ціни через серверні API-секрети, а фронтенд записує тільки підтверджену користувачем ціну.
- Схема Supabase (`inventory_items` та ін.) керується вручну через SQL-скрипти в `sklad/supabase/*.sql`, які користувач виконує в Supabase SQL Editor. **Якщо додаєш нову колонку і фронтенд одразу починає її використовувати в INSERT — попередь користувача, що без виконання SQL-міграції ця дія (напр. "Додати товар") почне падати з помилкою.** Так було з `003_add_internal_use_flag.sql`. Для цін INSERT не змінювали — без `004_add_price_tracking.sql` падає тільки збереження ціни, не додавання товару.

## Відомі обмеження / свідомо прийняті ризики

Щоб наступний аудит (людський чи AI) не "відкривав" це заново як несподівану знахідку — нижче задокументовані речі, які вже відомі й свідомо прийняті для поточного масштабу (один будинок, кілька співробітників), а не проґавлені баги:

- **RLS відкритий на insert/update для anon/authenticated** на "робочих" таблицях (`schedule`, `garbage`, `dispatcher`, `chat`, `photos`, `inventory_items`, `inventory_logs`, `inventory_receipts`, `inventory_audits`, `inventory_audit_items`) — `using(true)`/`with check(true)`. Anon-ключ вшитий у публічний HTML. Обґрунтування: маленька довірена команда, PIN-екран — єдиний бар'єр перед UI, не перед даними напряму. Якщо масштаб зросте (більше будинків/користувачів) — це треба переглянути (RPC-обгортки або Supabase Auth), але зараз свідомий компроміс, не проґавлена діра.
- **PIN — не per-user авторизація.** Один спільний PIN на весь застосунок (два — для журналу: вхід/скидання). Нема аудиту "хто саме зробив дію" — поля `author`/`issued_to`/`person` це вільний текст від користувача, не перевірена особа.
- **Telegram bot-токен зберігається в таблиці `telegram_config`**, не в Supabase Edge Function secrets — це свідомий вибір (простіше міняти chat_id/токен без редеплою функцій), не помилка. Але через це RLS на `telegram_config`/`osbb_telegram_config` **обов'язково має бути без жодної policy** (default-deny) — інакше токен читається anon-ключем. Так уже сталося один раз у видаленому проєкті журналу; при будь-якій зміні цих таблиць перевіряти, що policy не додались випадково.
- **Приклад-PIN у git — станом на липень 2026 підтверджено все ще не замінений на реальний** (жива база досі приймає той самий приклад-PIN, що прописаний у `sklad/supabase/001_setup_pin_auth.sql` і `005_merge_osbb_journal.sql` — самі цифри свідомо не повторюємо тут ще раз, вони й так у git-історії тих SQL-файлів, нема сенсу множити копії). Користувач свідомо відклав заміну ("поміняємо потім") — не забувати нагадувати, не намагатись замінити самому без прямого запиту (новий PIN не повинен з'являтись у логах сесії агента, міняти має сам користувач через SQL Editor).
- ~~Немає атомарності на списанні складу~~ — **виправлено в липні 2026**: `issue_item`/`receive_item` RPC (`sklad/supabase/006_atomic_stock_issue_receive.sql`) роблять `UPDATE ... WHERE quantity >= qty` і вставку логу в одній транзакції на сервері. `sklad/index.html` більше не читає залишок з клієнтського кешу для перевірки — викликає RPC і реагує на помилку `insufficient_stock`.

## Тестування в цьому пісочничному середовищі

- Вихідний трафік до `cdn.tailwindcss.com`, `fonts.googleapis.com`, Supabase заблокований проксі — **osbb/index.html (Tailwind-залежний) не рендериться повністю в headless-браузері тут**, sklad/index.html рендериться нормально (чиста CSS, без CDN-залежності для лейауту).
- Робочий спосіб візуальної перевірки: `python3 -m http.server` в корені репо + Playwright (`executablePath: '/opt/pw-browsers/chromium'`) + `page.addInitScript(() => sessionStorage.setItem('auth','ok'))` щоб проскочити PIN-екран. Playwright треба імпортувати з `/opt/node22/lib/node_modules` (глобальний пакет), інакше `Cannot find package 'playwright'`.
- Завжди запускати `node scripts/smoke-check.mjs` після правок — станом на липень 2026 це 176+ перевірок критичних RPC/PIN-flow/SW/shared-файлів; число росте з кожною правкою, не орієнтуватись на конкретну цифру, орієнтуватись на "усі мають бути ok".
- Через відсутність збірки легко зламати HTML мовчки (незакритий тег) — після великих правок перевіряти баланс тегів: `python3 -c "data=open('FILE').read(); print(data.count('<div'), data.count('</div>'))"` для div/svg/button/label.

## Відкриті питання на наступну сесію (липень 2026)

- **Multi-tenancy (`building_id`)** — користувач підняв питання, чи вводити поле для кількох будинків. **Не робити без явного запиту**: зараз RLS навмисно відкритий (`using(true)`) для anon в межах одного будинку (див. "Відомі обмеження" вище) — додати колонку `building_id` без реальної ізоляції (auth.uid() + членство в будинку замість спільного PIN) нічого не захистить, лише створить ілюзію multi-tenancy. Перш ніж починати — уточнити в користувача, чи справді з'являється другий будинок/клієнт, бо інакше це велика робота "про запас".
- **Storybook / візуальні тести** — вимагає компонентного фреймворку (React/Vue/Svelte) і build-step, що прямо суперечить "немає системи збірки" (див. "Що це за код" вище). Зараз немає окремих UI-компонентів, тільки величезні HTML-файли з дублюванням розмітки — Storybook самим собою нема для чого показувати. Реальний передумовний крок — набагато глибша компонентизація, тижні-місяці роботи, не "додати інструмент за раз".
- **PIN не замінений** — див. "Відомі обмеження" вище.

## Git-процес

- Кожна зміна — окрема гілка `claude/<щось>` від `main`, окремий PR через `mcp__github__create_pull_request`.
- Одразу підписуватись на PR (`subscribe_pr_activity`) і планувати self-check-in через `send_later` (~60 хв), якщо PR не змержено відразу.
- **Користувач попросив мержити PR одразу, не питаючи підтвердження** (July 2026) — це durable-інструкція для цього репозиторію, не разова.
- PR-шаблону в репо немає.
